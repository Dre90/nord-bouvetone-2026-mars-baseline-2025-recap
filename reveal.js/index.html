Baseline 2025

Web Platform features you can now use everywhere

Source: web.dev/baseline/2025

Welcome! Today we're looking at web platform features that became part of Baseline in 2025. These are features that now work across all major browsers — Chrome, Edge, Firefox, and Safari. The content is based on the official Baseline 2025 recap from web.dev.

What is Baseline?

Baseline gives you clear information about which web platform features are safe to use in your projects today.

A feature is Baseline when it works across all major browsers.

Baseline is an initiative by the W3C WebDX Community Group. A feature becomes "Baseline Newly available" once it ships in the latest stable version of all major browsers. After 30 months it moves to "Baseline Widely available". This gives developers a clear, trustworthy signal for when a feature is safe to adopt without polyfills.

CSS: Styling & Layout

  • @scope
  • content-visibility
  • scrollbar-color
  • print-color-adjust
  • font-family: math
Let's start with CSS features for styling and layout. We have five features here: scoped styling, a big performance win with content-visibility, cross-browser scrollbar theming, print control, and a math font family.

@scope

Scoped styling lets you limit CSS rules to a specific DOM subtree, avoiding class-name collisions and deep selectors.

@scope (.card) to (.card-content) {
  p { color: gray; }
}

MDN: @scope

@scope lets you constrain CSS rules to a subtree of the DOM. The "to" clause sets a lower boundary, so rules don't leak into nested components. This is a native alternative to BEM naming conventions or CSS Modules for avoiding style collisions. Especially useful in design systems where components are composed together.

scrollbar-color

Customize the thumb and track colors of scrollbars without non-standard pseudo-elements.

.container {
  scrollbar-color: #6969dd #e0e0e0;
}

MDN: scrollbar-color

Before this, styling scrollbars required WebKit-specific pseudo-elements that didn't work in Firefox. Now you can set the thumb and track color with a single standard property. The first value is the thumb color, the second is the track color. Great for dark-mode theming where default scrollbars look out of place.

content-visibility

Skips rendering of off-screen content, dramatically improving page load and scroll performance for long pages.

section {
  content-visibility: auto;
  contain-intrinsic-size: auto 500px;
}

web.dev: content-visibility

content-visibility: auto tells the browser to skip rendering of off-screen sections entirely. contain-intrinsic-size gives the browser a placeholder size so scrollbar height stays correct. This can improve initial render time dramatically on long pages — web.dev reported 7x rendering speedups in some cases. Became Baseline Newly available in late 2024/early 2025.

font-family: math

A generic font family for mathematical expressions, ensuring consistent rendering of math content across browsers.

math {
  font-family: math;
}

MDN: font-family

A new generic font family keyword specifically for mathematical notation. Browsers map it to an appropriate math font like Latin Modern Math or STIX Two Math. This ensures MathML content looks consistent across browsers without specifying exact font names.

print-color-adjust

Controls whether the browser may adjust colors and images when printing or saving to PDF.

.chart {
  print-color-adjust: exact;
}

MDN: print-color-adjust

By default browsers may strip background colors and images when printing to save ink. Setting print-color-adjust to exact tells the browser to preserve the original colors. Essential for charts, branded materials, and data visualizations that rely on color to convey meaning. This replaces the old non-standard -webkit-print-color-adjust.

CSS: Functions & Values

  • sign()
  • abs()
  • Single stop color gradients
Next up: new CSS math functions and a small but useful gradient improvement. These expand what you can do with pure CSS calculations without needing JavaScript.

sign()

Returns -1, 0, or 1 depending on the sign of the argument. Useful for direction-aware calculations in CSS.

.item {
  transform: translateX(
    calc(sign(var(--x)) * 100px)
  );
}

MDN: sign()

sign() returns -1, 0, or 1. In this example we extract the direction from a custom property and multiply it by a fixed distance. Useful for directional animations and layouts that need to respond to positive/negative values.

abs()

Returns the absolute value of a CSS math expression. Complements sign() for pure-CSS calculations.

.bar {
  width: abs(var(--value) * 1px);
}

MDN: abs()

abs() complements sign() — together they let you split a value into its magnitude and direction. Here we use it for a bar chart where the value can be negative but the width must be positive. Both sign() and abs() are part of the CSS Values Level 4 spec.

Single stop color gradients

Gradients with a single color stop are now valid, simplifying solid-color overlays and gradient fallbacks.

.overlay {
  background: linear-gradient(tomato);
}

Web Features Explorer

Previously, gradients required at least two color stops or they were invalid. Now a single-stop gradient is valid and renders as a solid color. This is handy for fallback patterns and generated backgrounds in CSS.

View Transitions

  • View Transition API
  • view-transition-class
  • Same-document view transitions
View Transitions are one of the most exciting additions to the platform. They let you animate between DOM states with smooth crossfade-like effects, natively. No more reach for FLIP libraries or JavaScript animation frameworks for basic transitions. Became Baseline Newly available in October 2025.

View Transition API

Enables smooth animated transitions between DOM states without JavaScript animation libraries.

::view-transition-old(root),
::view-transition-new(root) {
  animation-duration: 0.3s;
}

MDN: View Transition API

The View Transition API captures a snapshot of the old state, then cross-fades to the new state. You can customize the animation using standard CSS on the ::view-transition pseudo-elements. The browser handles the screenshot, overlay, and cleanup automatically.

view-transition-class

Assigns a class to a view transition pseudo-element, letting you style groups of transitioning elements together.

.card {
  view-transition-class: card;
}
::view-transition-group(.card) {
  animation-duration: 0.5s;
}

MDN: view-transition-class

Without view-transition-class you'd have to style each transitioning element individually by its view-transition-name. With this property you assign a class name, so you can write one CSS rule that matches all elements in that group. Perfect when you have a list of cards or grid items that should all animate the same way.

Same-document view transitions

Animate between states within a single page using document.startViewTransition().

document.startViewTransition(() => {
  updateDOM(); // swap content
});

web.dev: Same-document view transitions

Same-document view transitions became Baseline Newly available in October 2025. You call document.startViewTransition with a callback that makes DOM changes. The browser takes a snapshot, runs your callback, then animates the transition. This is the SPA use case — cross-document view transitions for MPAs are still being worked on.

HTML Features

  • Popover API
  • contenteditable="plaintext-only"
  • <link rel="dns-prefetch">
Three HTML features that reduce the need for JavaScript. The Popover API is probably the biggest one here — it replaces a ton of custom tooltip and dropdown code. We also have a refinement for contenteditable and a performance hint.

Popover API

A built-in way to create popovers, tooltips, and menus with automatic light-dismiss and top-layer rendering. No JavaScript needed for basic use.

<button popovertarget="mypopover">
  Open
</button>
<div id="mypopover" popover>
  Popover content!
</div>

web.dev: Popover API

The Popover API became Baseline Newly available in February 2025. It renders content in the top layer — above everything else, no z-index wars. Light dismiss means clicking outside automatically closes it. No JavaScript is needed for the basic toggle behavior — just the popover attribute and popovertarget. This replaces a huge amount of custom dropdown and tooltip JavaScript.

Demo: Popover API

Click the buttons — pure HTML, zero JavaScript

Live demo! Click the buttons to toggle popovers. Notice there's no JavaScript at all — just popovertarget on the button and popover on the div. Click outside the popover to dismiss it — that's light dismiss, built in. The popover renders in the top layer, so it appears above everything — no z-index needed.

contenteditable="plaintext-only"

Makes an element editable but strips all formatting on paste, keeping content as plain text.

<div contenteditable="plaintext-only">
  Type here…
</div>

web.dev: contenteditable plaintext-only

Became Baseline Newly available in March 2025. Before this, contenteditable would paste rich HTML which you then had to sanitize. With plaintext-only, the browser strips all formatting automatically. Advantage over textarea: it supports dynamic sizing, doesn't require a form, and can be styled more freely.

<link rel="dns-prefetch">

Tells the browser to resolve the DNS for a domain early, reducing latency when the user navigates to it.

<link rel="dns-prefetch"
      href="https://api.example.com">

MDN: dns-prefetch

dns-prefetch has been around for a while in some browsers, but it's now officially Baseline. DNS resolution can take 20–120 ms. Pre-resolving domains you know the user will need saves that delay. Use it for third-party APIs, CDNs, or analytics endpoints. It's free — the browser just does a DNS lookup with no actual connection.

Performance & Metrics

  • Largest Contentful Paint (LCP)
  • Event Timing API
  • scrollend event
Now we switch to JavaScript and Web APIs, starting with performance. Two Core Web Vitals APIs are now Baseline, plus a much-needed scroll event. LCP and Event Timing became Baseline in December 2025.

Largest Contentful Paint (LCP)

A Core Web Vital that measures loading performance — the time it takes for the largest visible element to render.

new PerformanceObserver((list) => {
  const entries = list.getEntries();
  const last = entries[entries.length - 1];
  console.log('LCP:', last.startTime);
}).observe({ type: 'largest-contentful-paint',
             buffered: true });

web.dev: LCP & INP

LCP is one of three Core Web Vitals used by Google for search ranking. A good LCP is under 2.5 seconds. The PerformanceObserver API lets you measure LCP in the field from real users. Both LCP and INP (Interaction to Next Paint) became Baseline in December 2025.

Event Timing API

Measures the latency of user interactions (click, keypress, etc.). Powers the Interaction to Next Paint (INP) metric.

new PerformanceObserver((list) => {
  for (const entry of list.getEntries()) {
    console.log(entry.name, entry.duration);
  }
}).observe({ type: 'event', buffered: true });

web.dev: Event Timing

Event Timing underpins the INP metric — Interaction to Next Paint. INP replaced FID (First Input Delay) as a Core Web Vital in March 2024. It measures the full latency from user input to the next frame being painted. A good INP is under 200 milliseconds.

scrollend event

Fires when scrolling has completed — no more guessing with debounced scroll listeners.

document.addEventListener('scrollend', () => {
  console.log('Scrolling finished!');
});

MDN: scrollend event

Before scrollend, developers had to debounce the scroll event and guess when scrolling stopped. Common patterns used 100–200ms timeouts which were unreliable. scrollend fires reliably when the scroll action is truly complete, including momentum scrolling on touch devices. Perfect for lazy-loading, snap-point detection, or analytics.

DOM & Interaction

  • Invoker Commands
  • document.caretPositionFromPoint()
  • dialog.requestClose()
  • Selection composed ranges
Four features that improve how we interact with the DOM. Invoker Commands extends the declarative pattern we saw with popovers. The other three are more specialized but solve real pain points for editor and dialog developers.

Invoker Commands

Declaratively wire buttons to actions on other elements — open dialogs, toggle popovers, play video — without JavaScript.

<button commandfor="my-dialog"
        command="show-modal">
  Open Dialog
</button>
<dialog id="my-dialog">Hello!</dialog>

MDN: Invoker Commands API

Invoker Commands let you connect a button to an action on another element using HTML attributes. The commandfor attribute points to the target element, and command specifies the action. Built-in commands include show-modal, close, toggle-popover, play, and pause. This follows the same "no-JS-needed" philosophy as the Popover API.

document.caretPositionFromPoint()

Returns the caret position (DOM node + offset) at given viewport coordinates. Useful for custom text editors and annotation tools.

document.addEventListener('click', (e) => {
  const pos = document.caretPositionFromPoint(
    e.clientX, e.clientY
  );
  console.log(pos.offsetNode, pos.offset);
});

MDN: caretPositionFromPoint()

Given x, y coordinates, this returns which text node and character offset the point lands on. Chrome previously had caretRangeFromPoint, but the standard API is caretPositionFromPoint. Now interoperable across all browsers. Essential for annotation tools, custom cursors, and text editors.

dialog.requestClose()

Requests closing a dialog, firing a cancel event first so the dialog can prevent closing (e.g. unsaved changes).

const dialog = document.querySelector('dialog');
// Fires 'cancel' event before closing
dialog.requestClose();

MDN: dialog.requestClose()

Unlike dialog.close() which closes immediately, requestClose() fires a cancel event first. You can call preventDefault() on the cancel event to keep the dialog open. Classic use case: "You have unsaved changes. Are you sure you want to close?" This also aligns with the behavior when a user presses Escape on a modal dialog.

Selection composed ranges

getComposedRanges() returns selection ranges that cross Shadow DOM boundaries — essential for web component editors.

const sel = window.getSelection();
const ranges = sel.getComposedRanges(
  shadowRoot
);

MDN: getComposedRanges()

The regular Selection API can't see inside Shadow DOM boundaries. getComposedRanges() solves this by returning ranges that span across shadow roots. You pass the shadow roots you want to include as arguments. Critical for rich text editors built with web components.

Data & Parsing

  • JSON import attributes
  • Unsanitized HTML parsing methods
  • URLPattern
  • Uint8Array base64 & hex conversion
Four features around data handling and parsing. JSON imports and URLPattern are probably the ones most of us can use right away. Uint8Array conversions are a nice quality-of-life improvement for binary data handling.

JSON import attributes

Import JSON files directly in JavaScript modules using the with syntax.

import data from './config.json'
  with { type: 'json' };

console.log(data.version);

web.dev: JSON imports

JSON import attributes became Baseline Newly available in June 2025. The "with" keyword tells the browser what type of module to expect. This replaces the older "assert" syntax from the original proposal. The JSON is imported as a default export — a plain JavaScript object. No fetch + parse needed.

Unsanitized HTML parsing methods

Document.parseHTMLUnsafe() and Element.setHTMLUnsafe() parse HTML strings including declarative Shadow DOM.

const doc = Document.parseHTMLUnsafe(
  htmlString
);
element.setHTMLUnsafe(htmlString);

MDN: parseHTMLUnsafe()

These methods parse HTML including declarative Shadow DOM templates. The "Unsafe" in the name means they don't sanitize the HTML — similar to innerHTML. The key difference from DOMParser is support for declarative shadow DOM. Use these when you need to hydrate server-rendered web components.

URLPattern

A built-in URL pattern matching API — like route matching in frameworks, but native.

const pattern = new URLPattern({
  pathname: '/users/:id'
});
pattern.test('https://example.com/users/42');
// true

web.dev: URLPattern

URLPattern became Baseline Newly available in September 2025. It uses the same syntax as path-to-regexp which powers Express.js and many other routers. You can match pathnames, hostnames, protocols, and more. Useful in Service Workers for route-based caching strategies, and in any URL routing logic.

Uint8Array base64 & hex

Convert between Uint8Array and base64/hex strings natively — no more btoa workarounds.

const bytes = Uint8Array.fromBase64('SGVsbG8=');
const hex = bytes.toHex(); // "48656c6c6f"
const b64 = bytes.toBase64(); // "SGVsbG8="

MDN: Uint8Array.fromBase64()

Before this, converting binary data to base64 required awkward workarounds with btoa/atob and String.fromCharCode. Now Uint8Array has native fromBase64(), toBase64(), fromHex(), and toHex() methods. Clean and efficient. Useful for cryptography, file handling, and binary protocols.

JavaScript Language

  • Iterator helpers
  • Promise.try()
  • RegExp.escape()
  • Float16Array
  • Atomics.waitAsync() & Atomics.pause()
Now we get to core JavaScript language features. Iterator helpers and Promise.try are the ones you'll probably use most often. Float16Array is niche but important for ML/GPU work. Atomics additions are for SharedArrayBuffer coordination.

Iterator helpers

Built-in .map(), .filter(), .take(), .drop() and more on all iterators — lazy and composable.

const evens = Iterator.from([1,2,3,4,5])
  .filter(n => n % 2 === 0)
  .toArray(); // [2, 4]

web.dev: Iterator helpers

Iterator helpers became Baseline Newly available in March 2025. They add .map(), .filter(), .take(), .drop(), .flatMap(), .reduce(), .forEach(), .some(), .every(), .find(), and .toArray(). Unlike Array methods, these are lazy — they only process elements as needed. Works on Map, Set, generators, and any iterable. A big ergonomic win.

Demo: Iterator Helpers

Click the buttons to run iterator examples live

Live demo! Three examples: 1. filter + take — shows lazy evaluation, only processes until 3 evens are found. 2. map on a Map iterator — demonstrates it works on any iterable, not just arrays. 3. Generator + drop — infinite Fibonacci sequence, take 5 values after skipping 5. Lazy eval makes this possible. All running natively in the browser — no polyfills!

Promise.try()

Wraps a sync or async function in a Promise so errors are always caught in the promise chain.

Promise.try(() => {
  return maybeSync();
}).then(result => {
  console.log(result);
});

web.dev: Promise.try

Became Baseline Newly available in January 2025. If the callback throws synchronously, it’s caught as a rejected Promise. If it returns a value, it’s wrapped in a resolved Promise. If it returns a Promise, it’s returned as-is. Perfect for APIs that accept either sync or async callbacks.

RegExp.escape()

Safely escapes a string for use inside a regular expression — no more hand-rolled escape functions.

const input = 'price: $5.00 (USD)';
const escaped = RegExp.escape(input);
// "price\\:\\ \\$5\\.00\\ \\(USD\\)"

MDN: RegExp.escape()

Every JavaScript developer has written a regex escape function at some point. Now there's a built-in. RegExp.escape() escapes all characters that have special meaning in regex. Useful when building regex from user input, like search-and-highlight features.

Float16Array

A typed array for 16-bit floating-point numbers. Useful for GPU data, ML models, and memory-constrained scenarios.

const f16 = new Float16Array([1.5, 2.5, 3.5]);
console.log(f16.byteLength); // 6

MDN: Float16Array

Float16Array stores 16-bit floats — half the size of Float32Array. Machine learning models commonly use float16 for weights and activation values. Also useful for GPU interop via WebGPU, which supports float16 natively. Trade-off: less precision and smaller range, but half the memory.

Atomics.waitAsync()

Non-blocking wait on a shared memory location — enables coordination between the main thread and workers without blocking.

const sab = new SharedArrayBuffer(4);
const i32 = new Int32Array(sab);
const result = Atomics.waitAsync(i32, 0, 0);
result.value.then(() => {
  console.log('Notified!');
});

MDN: Atomics.waitAsync()

Atomics.wait() blocks the thread, so it can't be used on the main thread. waitAsync() returns a Promise instead, so it works everywhere. This makes main-thread to worker coordination possible without polling. Useful for WASM-based applications and multi-threaded data processing.

Atomics.pause()

Hints to the CPU that a spin-wait loop is in progress, improving power efficiency in tight polling loops.

while (Atomics.load(i32, 0) === 0) {
  Atomics.pause();
}

MDN: Atomics.pause()

Atomics.pause() is a low-level hint for spin-wait loops. It tells the CPU "I'm in a busy-wait loop" so it can reduce power consumption and yield execution resources. Similar to the x86 PAUSE instruction. Very niche but important for high-performance concurrent code.

More Web APIs

  • ClipboardItem.supports()
  • Screen Wake Lock
  • Intl.DurationFormat
Three more web APIs to round things off. Clipboard feature detection, keeping the screen awake, and duration formatting.

ClipboardItem.supports()

Check if a MIME type is supported for clipboard read/write before attempting the operation.

if (ClipboardItem.supports('image/png')) {
  // safe to write PNG to clipboard
}

web.dev: ClipboardItem.supports()

Became Baseline Newly available in April 2025. Before this, you had to try/catch clipboard operations to see if a format was supported. Now you can feature-detect upfront. Browsers support different MIME types — this lets you adapt.

Screen Wake Lock

Prevents the screen from dimming or locking — great for recipe apps, presentations, and kiosk displays.

const wakeLock =
  await navigator.wakeLock.request('screen');

// Release when done
wakeLock.release();

MDN: Screen Wake Lock API

Prevents the screen from going to sleep while your app is active. Use cases: cooking apps showing a recipe, navigation apps, video calls, kiosk displays. The lock is automatically released when the tab becomes hidden or the document is unloaded. You should re-acquire it on visibilitychange if the user returns to the tab.

Intl.DurationFormat

Locale-aware formatting of durations — hours, minutes, seconds — without external libraries.

const fmt = new Intl.DurationFormat('en', {
  style: 'long'
});
fmt.format({ hours: 1, minutes: 30 });
// "1 hour, 30 minutes"

web.dev: Intl.DurationFormat

Became Baseline Newly available in March 2025. Formats durations according to locale rules — styles include long, short, narrow, and digital. No more hardcoding "hours" and "minutes" strings or using libraries like date-fns for this. Supports years, months, weeks, days, hours, minutes, seconds, milliseconds, microseconds, nanoseconds.

What's next? Baseline 2026

Features that have become part of Baseline so far in 2026:

HTML & CSS

  • rcap unit
  • rch unit

JavaScript & Web APIs

  • JS modules in service workers

web.dev/baseline/2026

Looking ahead — Baseline 2026 is already underway. On the CSS side, four new root-relative font units: rcap, rch, rex, and ric — similar to rem but for cap height, character width, x-height, and ideographic character width of the root element. On the JS side, you can now use JavaScript modules (import/export) in service workers, and the Navigation API provides a modern replacement for history.pushState with better lifecycle events. Check web.dev/baseline/2026 to follow along as more features land.
