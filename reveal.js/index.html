Baseline 2025

Webplattform-funksjoner du nå kan bruke overalt

Kilde: web.dev/baseline/2025

Velkommen! I dag skal vi gå gjennom de viktigste webfunksjonene som ble Baseline i 2025. Men først — hva betyr egentlig Baseline?

Hva er Baseline?

Baseline gjør det enkelt å vite hvilke webfunksjoner du trygt kan bruke i prosjektene dine i dag.

En funksjon er Baseline når den fungerer i alle store nettlesere.

Baseline er et initiativ fra W3C WebDX Community Group. Tanken er enkel: i stedet for å sjekke caniuse for hver nettleser, gir Baseline deg ett klart svar — fungerer dette overalt, eller ikke? Når en funksjon støttes i Chrome, Edge, Firefox og Safari, er den Baseline.

Se etter badge-ene på MDN

På MDN ser du Baseline-statusen som en badge øverst på siden. Den blå — «Newly available» — betyr at funksjonen nettopp har landet i alle nettlesere. Trygt å bruke, men noen brukere kan fortsatt ha eldre versjoner. Den grønne — «Widely available» — betyr at det har gått minst 30 måneder. Da kan du bruke den uten forbehold. Alt vi skal se i dag er funksjoner som fikk den blå badge-en i løpet av 2025.

CSS: Styling & Layout

CSS: Styling & Layout

  • @scope
  • content-visibility
  • scrollbar-color
  • print-color-adjust
  • font-family: math
Vi starter med CSS. Fem funksjoner for styling og layout. Den viktigste er nok content-visibility, som kan gi enorme ytelsesgevinster. Vi har også scoped styling med @scope, scrollbar-theming som endelig fungerer overalt, en utskriftsegenskap, og en egen fontfamilie for matte.

@scope

Begrens CSS-regler til en bestemt del av DOM-treet — unngå klassenavnkollisjoner og dype selektorer.

@scope (.card) to (.card-content) {
  p { color: gray; }
}

MDN: @scope

Med @scope kan du si «disse CSS-reglene gjelder bare innenfor dette undertreet». Du kan også sette en nedre grense med «to»-klausulen, så reglene ikke lekker inn i nestede komponenter. Tenk på det som et nativt alternativ til BEM eller CSS Modules — du slipper lange selektorer og navnekonvensjoner. Veldig nyttig i designsystemer der du setter sammen mange komponenter.

scrollbar-color

Stil scrollbarer med standard CSS — velg farge på tommel og spor uten proprietære pseudo-elementer.

.container {
  scrollbar-color: #6969dd #e0e0e0;
}

MDN: scrollbar-color

Før måtte du bruke WebKit-spesifikke pseudo-elementer for å style scrollbarer, og det fungerte ikke i Firefox. Nå er det én standard egenskap: første verdi er tommelfargen, andre er sporfargen. Særlig nyttig for dark mode, der de lyse standard-scrollbarene ser helt feil ut.

content-visibility

Lar nettleseren hoppe over rendering av innhold som ikke er synlig — gir dramatisk bedre ytelse på lange sider.

section {
  content-visibility: auto;
  contain-intrinsic-size: auto 500px;
}

web.dev: content-visibility

En skikkelig ytelsesvinner. Med content-visibility: auto sier du til nettleseren: «ikke bry deg med å rendre det som er utenfor skjermen.» contain-intrinsic-size gir en placeholder-størrelse, så scrollbaren oppfører seg riktig. web.dev rapporterte opptil 7x raskere rendering på lange sider. Ble Baseline sent i 2024, tidlig 2025.

font-family: math

Et generisk fontnøkkelord for matte — gir konsistent visning av matematisk innhold i alle nettlesere.

math {
  font-family: math;
}

MDN: font-family

Dette er et nytt generisk fontnøkkelord, på linje med serif og sans-serif. Nettleseren velger automatisk en passende matematikkfont, som Latin Modern Math eller STIX Two Math. Så MathML-innholdet ditt ser likt ut uansett hvilken nettleser brukeren har — uten at du trenger å spesifisere eksakte fontnavn.

print-color-adjust

Kontrollerer om nettleseren kan justere farger og bilder ved utskrift eller lagring til PDF.

.chart {
  print-color-adjust: exact;
}

MDN: print-color-adjust

Nettlesere fjerner ofte bakgrunnsfarger og bilder når du skriver ut, for å spare blekk. Med print-color-adjust: exact sier du «behold fargene mine som de er.» Viktig hvis du har diagrammer eller visualiseringer der farge betyr noe. Dette erstatter den gamle -webkit-print-color-adjust som ikke var standardisert.

CSS: Funksjoner & Verdier

CSS: Funksjoner & Verdier

  • sign()
  • abs()
  • Gradienter med ett fargestopp
Så har vi et par nye CSS-matematikkfunksjoner, pluss en liten gradient-forbedring. Disse gjør at du kan løse ting i ren CSS som før krevde JavaScript.

sign()

Returnerer -1, 0 eller 1 avhengig av fortegnet til argumentet. Nyttig for retningsavhengige beregninger i CSS.

.item {
  transform: translateX(
    calc(sign(var(--x)) * 100px)
  );
}

MDN: sign()

sign() gir deg -1, 0 eller 1 — bare fortegnet. Her i eksempelet henter vi retningen fra en custom property og ganger den med en fast avstand. Det er nyttig når du har animasjoner eller layouts som må vite «trekker vi til høyre eller venstre?» uten å bry seg om selve verdien.

abs()

Returnerer den absolutte verdien av et CSS-matematikkuttrykk. Komplementerer sign() for rene CSS-beregninger.

.bar {
  width: abs(var(--value) * 1px);
}

MDN: abs()

abs() og sign() hører sammen — den ene gir deg størrelsen, den andre retningen. I eksempelet bruker vi abs() for et stolpediagram der verdien kan være negativ, men bredden må alltid være positiv. Begge er en del av CSS Values Level 4.

Gradienter med ett fargestopp

Gradienter med bare ett fargestopp er nå gyldige, noe som forenkler ensfarget overlegg og gradient-fallbacks.

.overlay {
  background: linear-gradient(tomato);
}

Web Features Explorer

Før var en gradient med bare étt fargestopp ugyldig CSS — du måtte ha minst to. Nå er det lov, og resultatet er rett og slett en helfarge. Høres kanskje uvesentlig ut, men det er nyttig for fallback-mønstre og genererte bakgrunner.

View Transitions

View Transitions

  • View Transition API
  • view-transition-class
  • Same-document view transitions
View Transitions er kanskje det kuleste som ble Baseline i fjor. Kort forklart: nettleseren tar et øyeblikksbilde, gjør DOM-endringen, og crossfader mellom gammelt og nytt. Du slipper FLIP-biblioteker eller tunge animasjonsrammeverk. Ble Baseline i oktober 2025.

View Transition API

Gir deg jevne, animerte overganger mellom DOM-tilstander — uten JavaScript-animasjonsbiblioteker.

::view-transition-old(root),
::view-transition-new(root) {
  animation-duration: 0.3s;
}

MDN: View Transition API

Du styrer animasjonen med vanlig CSS på ::view-transition-old og ::view-transition-new pseudo-elementene. Nettleseren håndterer øyeblikksbildet, overlegget og oppryddingen — du trenger bare CSS.

view-transition-class

Tildeler en klasse til et view transition pseudo-element, slik at du kan style grupper av elementer i overgang sammen.

.card {
  view-transition-class: card;
}
::view-transition-group(.card) {
  animation-duration: 0.5s;
}

MDN: view-transition-class

Uten denne måtte du style hvert element individuelt via view-transition-name. Med view-transition-class kan du gruppere elementer, så du skriver én CSS-regel for hele gruppen. Superhendig når du har en liste med kort eller grid-elementer som alle skal animeres likt.

Same-document view transitions

Animer mellom tilstander på én og samme side med document.startViewTransition().

document.startViewTransition(() => {
  updateDOM(); // swap content
});

web.dev: Same-document view transitions

Du wrapper DOM-endringene dine i document.startViewTransition(), og nettleseren gjør resten. Merk at dette gjelder innenfor én side, altså SPA-er (Single Page Applications). Overganger mellom ulike sider — cross-document transitions for MPA-er (Multi Page Applications) — er fortsatt under utvikling.

Demo: View Transitions

Klikk på knappen for å se en same-document view transition i aksjon

La oss se det i aksjon! Klikk på knappen for å veksle mellom kompakt og utvidet kort. Se den jevne crossfade-animasjonen — nettleseren tar øyeblikksbildet, gjør DOM-endringen, og animerer. Både avataren og kortet har egne view-transition-name, så de animeres uavhengig av hverandre. Alt dette med bare document.startViewTransition() og CSS — null ekstra biblioteker.

HTML-funksjoner

HTML-funksjoner

  • Popover API
  • contenteditable="plaintext-only"
  • <link rel="dns-prefetch">
Tre HTML-funksjoner som gjør at du slipper å skrive JavaScript. Popover API er den store her — den erstatter haugevis av tooltip- og dropdown-kode. I tillegg har vi en forbedring for contenteditable og et ytelses-hint for DNS.

Popover API

Innebygde popovers, tooltips og menyer med light-dismiss og top-layer — uten JavaScript.

<button popovertarget="mypopover">
  Open
</button>
<div id="mypopover" popover>
  Popover content!
</div>

web.dev: Popover API

Popover API ble Baseline i februar 2025. Innholdet rendres i top layer — over absolutt alt annet. Ingen z-index-krig. Og light dismiss er innebygd: klikker du utenfor, lukkes den automatisk. Det fine er at alt dette fungerer med bare HTML-attributter — popover på elementet og popovertarget på knappen. Null JavaScript. Dette erstatter veldig mye kode som mange av oss har skrevet for hånd.

Demo: Popover API

Klikk på knappene — ren HTML, null JavaScript

La meg vise! Klikk på knappene for å se popovers i aksjon. Legg merke til: det er null JavaScript her. Bare popovertarget på knappen og popover på div-en. Klikk utenfor for å lukke — det er light dismiss, innebygd. Den rendres i top layer, så den vises over alt uten z-index.

contenteditable="plaintext-only"

Gjør et element redigerbart, men fjerner all formatering ved innliming, slik at innholdet forblir ren tekst.

<div contenteditable="plaintext-only">
  Type here…
</div>

web.dev: contenteditable plaintext-only

Ble Baseline i mars 2025. Dere kjenner sikkert problemet: du limer inn tekst fra nettet i et contenteditable-felt, og får med all stylingen. Med plaintext-only stripper nettleseren formateringen for deg automatisk. Og i motsetning til textarea støtter dette dynamisk størrelse og trenger ikke være i et skjema.

<link rel="dns-prefetch">

Ber nettleseren om å utføre DNS-oppslag for et domene tidlig, noe som reduserer ventetiden når brukeren navigerer dit.

<link rel="dns-prefetch"
      href="https://api.example.com">

MDN: dns-prefetch

dns-prefetch har fungert i noen nettlesere lenge, men nå er det offisielt Baseline. Et DNS-oppslag kan ta 20–120 ms, så hvis du vet at brukeren kommer til å trenge et domene — en API, en CDN — kan du be nettleseren gjøre oppslaget på forhånd. Det koster ingenting — det er bare et DNS-oppslag, ingen faktisk tilkobling.

Ytelse & Metrikker

Ytelse & Metrikker

  • Largest Contentful Paint (LCP)
  • Event Timing API
  • scrollend event
Nå bytter vi over til JavaScript og Web API-er. To Core Web Vitals API-er er nå Baseline, pluss en scroll-hendelse mange har ventet på. LCP og Event Timing ble Baseline i desember 2025.

Largest Contentful Paint (LCP)

En Core Web Vital som måler lasteytelse — tiden det tar før det største synlige elementet gjengis.

new PerformanceObserver((list) => {
  const entries = list.getEntries();
  const last = entries[entries.length - 1];
  console.log('LCP:', last.startTime);
}).observe({ type: 'largest-contentful-paint',
             buffered: true });

web.dev: LCP & INP

LCP er en av de tre Core Web Vitals som Google bruker i søkerangeringen. En god LCP er under 2,5 sekunder. Med PerformanceObserver kan du måle dette fra ekte brukere i produksjon. Både LCP og INP ble Baseline i desember 2025.

Event Timing API

Måler latensen til brukerinteraksjoner (klikk, tastetrykk osv.). Ligger bak Interaction to Next Paint (INP)-metrikken.

new PerformanceObserver((list) => {
  for (const entry of list.getEntries()) {
    console.log(entry.name, entry.duration);
  }
}).observe({ type: 'event', buffered: true });

web.dev: Event Timing

Event Timing er API-et bak INP — Interaction to Next Paint. INP erstattet FID som Core Web Vital i mars 2024, så dette er relevant for alle som bryr seg om ytelse. Det måler hele veien fra brukerinput til neste frame er rendret. En god INP er under 200 ms.

scrollend event

Utløses når scrolling er fullført — slutt på gjetting med debounced scroll-lyttere.

document.addEventListener('scrollend', () => {
  console.log('Scrolling finished!');
});

MDN: scrollend event

Dette er en dere har savnet. Før måtte man debounce scroll-eventen og gjette når scrollingen var ferdig. De fleste brukte 100–200 ms timeouts, og det var upålitelig. scrollend fyres når scrollingen faktisk er ferdig, inkludert momentum-scrolling på mobil. Super for lazy-loading, snap-point-deteksjon, eller analyse.

DOM & Interaksjon

DOM & Interaksjon

  • Invoker Commands
  • document.caretPositionFromPoint()
  • dialog.requestClose()
  • Selection composed ranges
Fire funksjoner som gjør det enklere å jobbe med DOM-en. Invoker Commands bygger videre på det deklarative mønsteret vi akkurat så med popovers. De tre andre er mer spesialiserte — men løser reelle problemer for de som bygger editorer eller dialoger.

Invoker Commands

Koble knapper deklarativt til handlinger på andre elementer — åpne dialoger, veksle popovers, spille video — uten JavaScript.

<button commandfor="my-dialog"
        command="show-modal">
  Open Dialog
</button>
<dialog id="my-dialog">Hello!</dialog>

MDN: Invoker Commands API

Samme tankegang som Popover API: du beskriver handlingen i HTML, uten JavaScript. commandfor peker på målelementet, command sier hva som skal skje. Det finnes innebygde kommandoer som show-modal, close, toggle-popover, play og pause. Spår at vi kommer til å se mye mer av dette mønsteret fremover.

document.caretPositionFromPoint()

Returnerer markørposisjonen (DOM-node + offset) ved gitte viewport-koordinater. Nyttig for egendefinerte tekstredigerere og annotasjonsverktøy.

document.addEventListener('click', (e) => {
  const pos = document.caretPositionFromPoint(
    e.clientX, e.clientY
  );
  console.log(pos.offsetNode, pos.offset);
});

MDN: caretPositionFromPoint()

Denne gir deg tekstnoden og tegnposisjonen for et gitt punkt på skjermen. Chrome hadde caretRangeFromPoint før, men nå er det standard-API-et som fungerer i alle nettlesere. Relevant for de som bygger annotasjonsverktøy, egne markører eller tekstredigerere.

dialog.requestClose()

Ber om å lukke en dialog, og utløser en cancel-hendelse først slik at dialogen kan forhindre lukking (f.eks. ulagrede endringer).

const dialog = document.querySelector('dialog');
// Fires 'cancel' event before closing
dialog.requestClose();

MDN: dialog.requestClose()

Forskjellen fra dialog.close() er at requestClose() gir dialogen en sjanse til å si nei. Den fyrer en cancel-hendelse først, og du kan kalle preventDefault() for å holde dialogen åpen. Det klassiske eksempelet: «Du har ulagrede endringer — vil du virkelig lukke?» Samme oppførsel som når brukeren trykker Escape på en modal.

Selection composed ranges

getComposedRanges() returnerer seleksjonsområder som krysser Shadow DOM-grenser — viktig for tekstbehandlere bygget med webkomponenter.

const sel = window.getSelection();
const ranges = sel.getComposedRanges(
  shadowRoot
);

MDN: getComposedRanges()

Det vanlige Selection API-et stopper ved Shadow DOM-grensen — du kan ikke se inn. getComposedRanges() løser det ved å gi deg seleksjoner som går på tvers av shadow roots. Du sender inn de shadow roots du vil inkludere. Viktig for alle som bygger rike tekstredigerere med webkomponenter.

Data & Parsing

Data & Parsing

  • JSON import attributes
  • HTML-parsing uten sanitering
  • URLPattern
  • Uint8Array base64- & hex-konvertering
Fire funksjoner for datahåndtering og parsing. JSON-imports og URLPattern er de mest hverdagslige — de kan mange av dere ta i bruk med en gang. Uint8Array-konverteringene er en velkommen forbedring for de som jobber med binærdata.

JSON import attributes

Importer JSON-filer direkte i JavaScript-moduler med with-syntaksen.

import data from './config.json'
  with { type: 'json' };

console.log(data.version);

web.dev: JSON imports

Ble Baseline i juni 2025. Med «with»-nøkkelordet forteller du nettleseren at dette er JSON, ikke JavaScript. Dette erstatter den eldre «assert»-syntaksen fra det opprinnelige forslaget. Resultatet er et vanlig JavaScript-objekt som default eksport — ingen fetch og parse nødvendig.

HTML-parsing uten sanitering

Document.parseHTMLUnsafe() og Element.setHTMLUnsafe() parser HTML-strenger inkludert deklarativ Shadow DOM.

const doc = Document.parseHTMLUnsafe(
  htmlString
);
element.setHTMLUnsafe(htmlString);

MDN: parseHTMLUnsafe()

Disse to metodene parser HTML inkludert deklarative Shadow DOM-maler. «Unsafe» i navnet betyr bare at de ikke saniterer — litt som innerHTML. Forskjellen fra DOMParser er at disse støtter deklarativ Shadow DOM. Relevant når du skal hydrere serverrendrede webkomponenter.

URLPattern

Et innebygd API for URL-mønstermatching — som rutematching i rammeverk, men nativt.

const pattern = new URLPattern({
  pathname: '/users/:id'
});
pattern.test('https://example.com/users/42');
// true

web.dev: URLPattern

Ble Baseline i september 2025. Dette er rutematching slik dere kjenner det fra Express.js og lignende — samme path-to-regexp-syntaks, men innebygd i nettleseren. Du kan matche stier, vertsnavn, protokoller og mer. Særlig nyttig i Service Workers for rutebasert caching.

Uint8Array base64 & hex

Konverter mellom Uint8Array og base64/hex-strenger nativt — ingen flere btoa-workarounds.

const bytes = Uint8Array.fromBase64('SGVsbG8=');
const hex = bytes.toHex(); // "48656c6c6f"
const b64 = bytes.toBase64(); // "SGVsbG8="

MDN: Uint8Array.fromBase64()

De som har jobbet med binærdata i JavaScript vet hvor klønete det var — btoa, atob, String.fromCharCode... Nå har Uint8Array innebygde metoder: fromBase64(), toBase64(), fromHex() og toHex(). Rent og greit. Nyttig for kryptografi, filhåndtering og binære protokoller.

JavaScript-språket

JavaScript-språket

  • Iterator helpers
  • Promise.try()
  • RegExp.escape()
  • Float16Array
  • Atomics.waitAsync() & Atomics.pause()
Så kommer vi til selve JavaScript-språket. Iterator helpers og Promise.try er nok de dere får mest bruk for i hverdagen. Float16Array er mer nisje, men viktig om du jobber med ML eller GPU. Atomics-greiene er for de som driver med SharedArrayBuffer og multi-threading.

Iterator helpers

Innebygde .map(), .filter(), .take(), .drop() og mer på alle iteratorer — late og komponerbare.

const evens = Iterator.from([1,2,3,4,5])
  .filter(n => n % 2 === 0)
  .toArray(); // [2, 4]

web.dev: Iterator helpers

Ble Baseline i mars 2025. Dere får .map(), .filter(), .take(), .drop(), .flatMap(), .reduce() og en haug til — direkte på alle iteratorer. Og i motsetning til Array-metoder er disse late — de prosesserer bare elementer når du ber om dem. Fungerer på Map, Set, generatorer, alt som er iterable. En skikkelig oppgradering.

Promise.try()

Pakker inn en synkron eller asynkron funksjon i et Promise, slik at feil alltid fanges i promise-kjeden.

Promise.try(() => {
  return maybeSync();
}).then(result => {
  console.log(result);
});

web.dev: Promise.try

Ble Baseline i januar 2025. Den dekker tre tilfeller: – Kaster callbacken synkront? Du får et avvist Promise. – Returnerer en vanlig verdi? Du får et oppfylt Promise. – Returnerer et Promise? Det sendes rett gjennom. Hendig i API-er der du ikke vet om callbacken er synkron eller asynkron.

RegExp.escape()

Gjør en streng trygg for bruk i regulære uttrykk — slipper å skrive egne escape-funksjoner.

const input = 'price: $5.00 (USD)';
const escaped = RegExp.escape(input);
// "price\\:\\ \\$5\\.00\\ \\(USD\\)"

MDN: RegExp.escape()

Hvor mange av dere har skrevet en egen regex escape-funksjon? Sikkert de fleste. Nå finnes det en innebygd. Den escaper alle tegn som har spesiell betydning i regex. Hendig når du bygger regex fra brukerinput, for eksempel søk-og-uthev.

Float16Array

Et typet array for 16-bits flyttall. Nyttig for GPU-data, ML-modeller og minnesparende scenarier.

const f16 = new Float16Array([1.5, 2.5, 3.5]);
console.log(f16.byteLength); // 6

MDN: Float16Array

Float16Array er halvparten av størrelsen til Float32Array — 16-bits flyttall. Maskinlæringsmodeller bruker ofte float16 for vekter og aktiveringer, så dette er relevant for ML-greier i nettleseren. Også nyttig for WebGPU, som støtter float16 nativt. Du får mindre presisjon og verdiområde, men halvparten av minnet.

Atomics.waitAsync()

Asynkron venting på en delt minnelokasjon — muliggjør koordinering mellom hovedtråden og workers.

const sab = new SharedArrayBuffer(4);
const i32 = new Int32Array(sab);
const result = Atomics.waitAsync(i32, 0, 0);
result.value.then(() => {
  console.log('Notified!');
});

MDN: Atomics.waitAsync()

Problemet med Atomics.wait() er at den blokkerer tråden, så den kan ikke brukes på hovedtråden. waitAsync() løser det ved å returnere et Promise i stedet. Så du kan koordinere mellom hovedtråd og workers uten polling. Relevant for WASM-baserte apper og flertrådet databehandling.

Atomics.pause()

Gir CPU-en et hint om at en spin-wait-løkke pågår, og forbedrer strømeffektiviteten i tette polling-løkker.

while (Atomics.load(i32, 0) === 0) {
  Atomics.pause();
}

MDN: Atomics.pause()

Atomics.pause() er et lavnivå-hint for spin-wait-løkker. Det forteller CPU-en «jeg er i en busy-wait-løkke» slik at den kan redusere strømforbruket og frigjøre ressurser. Ligner på x86 PAUSE-instruksjonen. Veldig nisje, men viktig for høyytelses samtidig kode.

Flere Web API-er

Flere Web API-er

  • ClipboardItem.supports()
  • Screen Wake Lock
  • Intl.DurationFormat
Tre API-er til, så runder vi av. Funksjonsdeteksjon for utklippstavlen, holde skjermen våken, og varighetsformatering.

ClipboardItem.supports()

Sjekk om en MIME-type støttes for lesing/skriving til utklippstavlen før du forsøker operasjonen.

if (ClipboardItem.supports('image/png')) {
  // safe to write PNG to clipboard
}

web.dev: ClipboardItem.supports()

Ble Baseline i april 2025. Før måtte du prøve og feile med try/catch for å se om et format var støttet. Nå kan du spørre på forhånd. Nettlesere støtter ulike MIME-typer, så dette lar deg tilpasse deg.

Screen Wake Lock

Forhindrer at skjermen dimmes eller låses — flott for oppskriftsapper, presentasjoner og kiosk-skjermer.

const wakeLock =
  await navigator.wakeLock.request('screen');

// Release when done
wakeLock.release();

MDN: Screen Wake Lock API

Holder skjermen på så lenge appen din er aktiv. Tenk matlagingsapper, navigasjon, videosamtaler, kiosk-skjermer — alt der skjermen ikke må slå seg av. Låsen frigis automatisk når fanen blir skjult. Husk å hente den på nytt i visibilitychange-eventen hvis brukeren kommer tilbake.

Intl.DurationFormat

Språktilpasset formatering av varigheter — timer, minutter, sekunder — uten eksterne biblioteker.

const fmt = new Intl.DurationFormat('en', {
  style: 'long'
});
fmt.format({ hours: 1, minutes: 30 });
// "1 hour, 30 minutes"

web.dev: Intl.DurationFormat

Ble Baseline i mars 2025. Den formaterer varigheter etter språkregler — du kan velge mellom long, short, narrow og digital. Slipper å hardkode «timer» og «minutter» eller dra inn date-fns bare for dette. Støtter alt fra år ned til nanosekunder.
